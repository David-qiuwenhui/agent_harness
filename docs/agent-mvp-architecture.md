# Agent Harness — ToB 财经 SaaS Agent MVP 架构方案

## 1. 项目背景

为以电子流为主的财经 SaaS 系统搭建 Agent 能力，提供三大核心功能：

- **智能问答**：基于电子流页面上下文的实时问答
- **工作流自动化**：审批、流转、通知等流程的自动化编排
- **数据分析与洞察**：财务数据查询、报表生成、趋势分析

GitHub 仓库：https://github.com/David-qiuwenhui/agent_harness

## 2. 开源方案调研：learn-claude-code

### 2.1 项目概述

| 项目 | 信息 |
|------|------|
| 仓库 | shareAI-lab/learn-claude-code |
| Stars | 63K+ |
| 语言 | Python |
| 定位 | 教学型项目，从零搭建类 Claude Code 的 agent 框架 |
| 结构 | 20 个渐进式阶段（s01-s20），完整实现约 500 行 |

### 2.2 核心架构模式

```
User -> Agent Loop -> LLM -> Tool Dispatch -> Execute -> Feed Back -> Loop
```

Agent 的本质是一个循环：LLM 调用工具、拿到结果、继续推理，直到认为任务完成。

### 2.3 值得复用的模式

| 模式 | 说明 | 财经场景适配性 |
|------|------|----------------|
| Agent Loop | `while stop_reason == "tool_use"` 循环 | 通用，所有 agent 框架的基石 |
| Tool Dispatch | 统一的工具定义与分发机制 | 高，替换为财经业务工具即可 |
| Skill System | 动态加载领域知识（SKILL.md） | 高，可用于加载审批规则、财务制度 |
| Subagent | 子代理委托模式 | 中，可用于并行处理多个电子流 |
| Context Compact | 两层压缩（micro + auto） | 高，长对话场景必备 |
| Team/Messaging | 多 agent 协作的 message bus | 中，复杂审批链路可能需要 |

### 2.4 为什么不直接 fork

| 问题 | 说明 |
|------|------|
| 无企业级能力 | 缺少认证、权限、审计、多租户 |
| 单进程同步 | 不适合生产环境的高并发场景 |
| 工具层不匹配 | 面向编码任务（bash/read/write），不是财经业务 |
| 与业务系统割裂 | 没有和电子流系统的集成点 |

**结论**：学思想，不学实现。用 1-2 天跑通 s01-s06 理解核心机制即可。

## 3. 推荐的 MVP 架构

### 3.1 整体架构

```
|  | 认证/鉴权 |  | 会话管理  |  | 审计日志          |  |
|  +----------+  +----------+  +-------------------+  |
+----------------------+------------------------------+
                       |
                       v
+-----------------------------------------------------+
|                 Agent Core (核心引擎)                  |
|                                                      |
|  +----------------+  +----------------------------+  |
|  |  Agent Loop     |  |  Context Manager           |  |
|  |  (循环推理)     |  |  (上下文压缩 + 会话历史)    |  |
|  +----------------+  +----------------------------+  |
|  +----------------+  +----------------------------+  |
|  |  Tool Dispatch  |  |  Skill Loader              |  |
|  |  (工具分发)     |  |  (领域知识动态加载)         |  |
|  +----------------+  +----------------------------+  |
+----------------------+------------------------------+
                       |
          +------------+------------+
          v            v            v
   +------------+ +---------+ +----------+
   | 电子流工具  | | 财务工具 | | RAG 工具  |
   |            | |         | |          |
   | - 查询流程  | | - 查数据 | | - 文档检索|
   | - 审批操作  | | - 生成表 | | - 政策问答|
   | - 流转通知  | | - 趋势   | | - 知识库  |
   +------------+ +---------+ +----------+
          |            |            |
          v            v            v
   +-----------------------------------------+
   |         业务系统 / 数据源                 |
   |   (电子流引擎 / 财务数据库 / 知识库)      |
   +-----------------------------------------+
                       |
                       v
              +-----------------+
              |       LLM       |
              |  Claude / 国内   |
              +-----------------+
```

### 3.2 核心组件说明

#### Agent Gateway（API 层）

负责对外暴露接口，处理认证、会话和审计：

- **认证/鉴权**：对接现有 SaaS 的用户体系，确保 agent 操作有权限
- **会话管理**：维护多轮对话上下文，支持会话恢复
- **审计日志**：记录 agent 的每一次操作，满足金融合规要求

#### Agent Core（核心引擎）

从 learn-claude-code 提炼的核心模式：

- **Agent Loop**：LLM 推理 -> 工具调用 -> 结果反馈 -> 继续推理
- **Context Manager**：上下文窗口管理，长对话自动压缩
- **Tool Dispatch**：统一的工具注册与分发机制
- **Skill Loader**：动态加载财经领域知识（审批规则、财务制度等）

#### 业务工具层

替换 learn-claude-code 的编码工具，面向财经场景：

| 工具类别 | 工具名称 | 功能 |
|----------|----------|------|
| 电子流 | `query_workflow` | 查询电子流状态、详情、历史 |
| 电子流 | `approve_step` | 执行审批操作（通过/驳回） |
| 电子流 | `notify_stakeholder` | 向相关人员发送通知 |
| 财务 | `query_financial_data` | 查询财务数据（科目余额、明细等） |
| 财务 | `generate_report` | 生成财务报表 |
| 财务 | `analyze_trend` | 趋势分析 |
| 知识 | `search_documents` | RAG 检索内部文档 |
| 知识 | `query_policy` | 查询财务政策与制度 |

## 4. MVP 分阶段计划

### Phase 1：最小可用的 Agent（2 周）

```
目标：单场景 agent -- 电子流智能问答

范围：
- Agent Loop + Tool Dispatch 核心实现
- 1 个工具：query_workflow（查询电子流）
- 基本的 API 接口（REST）
- 简单的上下文管理

验收标准：
- 用户能用自然语言查询电子流状态
- Agent 能正确调用工具并返回结果
```

### Phase 2：扩展工具 + Skill（2 周）

```
目标：多工具 + 领域知识

范围：
- 增加 approve_step、query_financial_data 工具
- Skill 系统实现（加载审批规则、财务制度）
- 上下文压缩（借鉴 learn-claude-code 的两层压缩）
- 前端 Chat Widget 集成

验收标准：
- 支持审批操作（确认后执行）
- 能根据制度规则给出合规建议
```

### Phase 3：生产化（持续迭代）

```
目标：可上线的企业级 Agent

范围：
- 认证/鉴权集成
- 审计日志
- 多租户隔离
- 可观测性（链路追踪、指标监控）
- 性能优化（并发、缓存）

验收标准：
- 通过安全审计
- 压测达标
- 灰度上线
```

## 5. 启动前需确认的决策点

以下问题直接影响技术选型和架构设计，启动前必须明确：

| # | 决策项 | 选项 | 影响 |
|---|--------|------|------|
| 1 | **现有 SaaS 技术栈** | Java / Go / Node？前端 React / Vue？ | 决定 Agent 服务端语言、集成方式、前端 SDK |
| 2 | **LLM 选型** | Claude API / 国内模型（DeepSeek/Qwen） / 兼容多家？ | 决定 SDK 选型、是否需要模型路由层、数据合规 |
| 3 | **MVP 首发场景** | 智能问答 / 工作流自动化 / 数据分析？ | 决定首批工具实现、优先级排序 |
| 4 | **部署方式** | 私有化 / 云端 / 混合？ | 决定基础设施选型、数据安全方案 |
